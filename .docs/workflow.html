Process · commit-log-style task flow

Workflow

A lightweight, one-ticket-at-a-time flow for an agent to pick up and complete tasks. One ticket in, one log line out — then stop and wait for approval.

①

Layout

.docs/
  workflow.html            # this file
  tasks/
    status.html            # the pointer: what's next + the log
    bruno-testing-plan.html# how to write the Bruno tests
    assets/tasks.css       # shared styling for these notes
    backlog/               # tickets waiting to be picked up (.html)
    done/                  # completed tickets (.html)
②

How an agent works a task

  1. Read status.html. The Next pointer is the only thing you need to know which ticket to pick.
  2. Open that ticket in backlog/.
  3. Do the work described — code + its Bruno tests (follow bruno-testing-plan.html). Leave the commit working and incremental.
  4. Capture handoff notes — only if there are any. An issue, a decision worth flagging, or something a later task should assume → append a Notes section at the bottom of the ticket. Nothing to hand off? Skip it.
  5. Move the ticket from backlog/ to done/.
  6. Update status.html: add one log row for what you finished (mark [notes] if you left any), and repoint Next to the following backlog ticket.
  7. Stop and wait for approval. Do not start the next ticket.
③

Conventions

  • One agent, one ticket at a time. Finish, move, log, stop.
  • Notes are for the next agent — not a status report. Only what changes how the next task is done: issues, decisions, gotchas.
  • done/ is history. Once a ticket lands there, it isn't edited again.
  • Folders are the source of truth for state; status.html is the source of truth for "what now." If they disagree, trust the folders and fix status.html.
  • Each commit must be working and incremental — no commits that only add empty folders or .gitkeep placeholders.
