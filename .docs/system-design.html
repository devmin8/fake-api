01

Goal & non-goals

One backend you spin up once and point any frontend project at. It behaves like a real product API — sign-in, ownership, relational reads, live updates — so the frontend you build against it can be as complex as the real thing. The differentiator over JSONPlaceholder & friends: it actually accepts writes, enforces auth, models relationships, and streams a live feed over WebSockets.

In scope

  • A social/Q&A feed domain (StackOverflow × Twitter): users, posts with optional images, threaded comments, likes, tags, follows, bookmarks, notifications.
  • Real auth with a public/authenticated route split that mirrors StackOverflow — browse & subscribe freely; write only when signed in.
  • Cursor-paginated, filterable, sortable reads over genuinely relational data.
  • WebSocket topics for live counts, new posts, comments, and per-user notifications.
  • A toggleable firehose simulator — bot users that post/comment/like on an interval and broadcast events, so the feed visibly streams with no second client.

Non-goals

  • Not production-hardened. No multi-tenant, no horizontal scaling, no rate-limit infra beyond a courtesy limiter. Single-process is fine.
  • No email/SMS delivery, no OAuth providers (local credential auth only).
  • Not a teaching example of microservices — deliberately one modular monolith.
  • Backend-only repository. Any frontend behaviour described elsewhere (e.g. the companion feed guide covering the new-posts banner and client cursor handling) is learning reference for the human only — it is not to be implemented in this project. This repo ships an API; it has no UI.

Data model

IDs are ULIDs (time-sortable — they double as pagination cursors). All timestamps are ISO-8601 UTC. Counts on posts / comments are denormalized and updated in the same transaction as the action that changes them, so live count broadcasts are cheap and reads never aggregate.

Relationship notes

  • Posts ↔ Comments 1:N; comments self-reference via parent_id for one-level (or deep) threads.
  • Likes are polymorphic over posts and comments via (target_type, target_id) with a uniqueness guard so a user can't double-like.
  • Tags are M:N through post_tags.
  • Follows drive the personalized /feed; notifications are written whenever someone likes/comments/follows you (real users and bots).
  • FKs use ON DELETE CASCADE where a child can't outlive its parent (comments, likes, post_tags).
05

Auth & sessions

Cookie-based, server-set, never readable by JS. JWT is only the access-token format; storage is httpOnly cookies, so XSS can't exfiltrate a session.

CookieContentsFlagsTTL
access_tokenJWT, payload { sub: userId }httpOnly · Secure · SameSite=Lax · Path=/~15 min
refresh_tokenopaque random; stored hashed in refresh_tokenshttpOnly · Secure · SameSite=Lax · Path=/auth/refresh~7 days
csrf_tokenrandom; readable by JS (double-submit)Secure · SameSite=Laxsession

Flow

  1. POST /auth/register or /auth/login → verify with Bun.password → set all three cookies, return the user object.
  2. Every request: guard reads access_token, verifies JWT, attaches user to context. Missing/expired → 401.
  3. POST /auth/refresh (only endpoint the refresh cookie is sent to): validate against stored hash, rotate (issue new refresh, revoke old), re-issue access.
  4. POST /auth/logout: revoke refresh row, clear all cookies.
CSRF Double-submit: client reads csrf_token cookie and echoes it in an X-CSRF-Token header on every POST/PATCH/DELETE; server compares. Gated behind CSRF_ENABLED so it can be switched off while you're focused on learning a frontend, on when you want to practice the full handshake.
Cross-origin in dev Frontend (app.social.localhost) and API (api.social.localhost) are same-site, different-origin. CORS must echo the exact origin (never *) with Access-Control-Allow-Credentials: true; the frontend must send credentials:"include". Because they share the parent domain, SameSite=Lax cookies still attach — no need to weaken to None.

Public vs authenticated

Reading anything — feed, post detail, comments, profiles, tags — and subscribing to live counts is fully public, exactly like browsing StackOverflow signed-out. Creating, editing, liking, commenting, following, bookmarking, and reading your notifications require a session. Ownership-bound mutations (edit/delete) additionally check the row belongs to the caller.

07

Conventions

Pagination — directional cursor (two anchors)

All lists take ?limit (default 20, max 100), ?cursor, and ?dir=older|newer (default older). A cursor is an opaque base64 of created_at|id. Because ULIDs sort by time, a cursor is a stable boundary: inserts at the head can never shift what "older than X" means. This is the single property that lets a live feed paginate without gaps or dupes.

The two directions are independent and exist so a client can hold two anchors — the newest item it has (head) and the oldest (tail):

  • dir=older&cursor=<tail> — backfill (infinite scroll). Returns items strictly older than the tail, newest-first.
  • dir=newer&cursor=<head> — catch-up. Returns items strictly newer than the head. Critical contract: order these adjacent-to-head first (ascending from the cursor) so the page always butts directly against what the client already holds. If more exist than limit, the leftover overflow stays above the returned page (toward now) — never stranded in the middle.

So both edges are anchored to a row the client already owns; contiguity is structural, not best-effort. Fetching "the latest N" instead (absolute, unanchored) is what opens gaps — the API deliberately keys off the caller's cursor instead.

// collection response envelope — both boundary cursors returned
{
  "data":        [ ... ],
  "newestCursor": "01J…",        // head of this page
  "oldestCursor": "01J…",        // tail of this page
  "hasMore":     true          // more in the requested direction
}
Large-gap escape hatch /posts/updates?since=<head> returns the count of newer items. The client decides what to do with it; the server just reports the number. If a client requests dir=newer and hasMore stays true across pages, that's the signal a huge backlog accrued (long absence) — the client may prefer a fresh top-of-feed load over stitching. Both are plain list calls; no special endpoint needed.

Sorting

new (created_at desc · default) · top (like_count desc) · trending (likes & comments weighted, decayed by age). Directional cursors apply to new (chronological); top/trending are ranked, so they paginate older-only within a frozen ordering and don't take dir=newer.

Errors — one envelope

{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "body must be at least 1 character",
    "details": { "field": "body" }   // optional
  }
}

Codes map to status: VALIDATION_ERROR 422 · UNAUTHENTICATED 401 · FORBIDDEN 403 · NOT_FOUND 404 · CONFLICT 409 · RATE_LIMITED 429. Single resources return the object directly; collections use the envelope above.

08

Realtime / WebSocket

One endpoint, wss://api.social.localhost/ws, via Elysia .ws(). Clients subscribe to topics; the server pushes JSON events. Public topics need no session; the per-user topic requires a valid access-token cookie at upgrade.

Client → server

{ "action": "subscribe",   "topic": "post:01J…" }
{ "action": "unsubscribe", "topic": "feed:global" }

Server → client

{
  "topic": "post:01J…",
  "event": "post.liked",
  "data":  { "postId": "01J…", "likeCount": 42 },
  "ts":    "2026-06-23T12:00:00.000Z"
}
TopicAccessEvents
feed:globalPUBLICpost.created
tag:{slug}PUBLICpost.created
post:{id}PUBLICpost.liked · post.unliked · post.viewed · comment.created · comment.liked
user:{id}AUTHnotification.created

The WS hub is a small in-memory map of topic → Set<socket>. Any code path that mutates state (a real request handler or the simulator) calls hub.broadcast(topic, event, data) after its DB transaction commits — so the wire is always consistent with the database.

Signal, not source of truth Treat post.created as a notification that newer content exists, not as the authoritative payload. The reliable path is: socket says "something's new" → client re-fetches via GET /posts?dir=newer&cursor=<head>. This makes dropped/reconnected sockets harmless — on reconnect the client runs that one cursor query to reconcile anything missed while disconnected. The DB (via the cursor) is truth; the socket only decides when to ask. The fat-payload variant (trust the pushed post directly) is fine for this mock too, but the cursor refetch is the gap-proof one and the reason the API returns boundary cursors on every page.
09

Firehose simulator

This is what makes the feed feel alive without you opening a second browser. A timer fires every SIM_INTERVAL_MS; each tick performs N weighted random actions as bot users (is_bot=true, seeded up front), writing through the same service layer real requests use and emitting the same WS events.

ActionDefault weightEffect & broadcast
Create post20%New post (faker text, optional placeholder image, random tags) → post.created on feed:global + tag topics.
Comment35%Comment on a recent post → comment.created on post:{id}.
Like post30%Like a recent post → post.liked + new count.
Like comment / follow15%Like a comment or follow a user → relevant event + notification.

Controlled at runtime via /sim/*. Config: { enabled, intervalMs, actionsPerTick, weights }. Starts automatically when SIM_ENABLED=true. Because bots act through the real write path, every simulated like/comment lands in SQLite and shows up identically on next page load — no special-casing in your frontend.

10

Media uploads

POST /media accepts a multipart file, validates mime (images only) and size, writes it under ./data/uploads/<ulid>.<ext>, records a media row, and returns { url }. The returned URL is used as a post's image_url. Bot-generated posts use placeholder image services so the feed has imagery out of the box, giving you image rendering, lazy-loading, and aspect-ratio handling to practice against.

    auth-guard.ts     # derive user; .auth() + .owner() macros
  routes/
    auth.ts  users.ts  posts.ts  comments.ts
    tags.ts  bookmarks.ts  notifications.ts  media.ts  sim.ts
  services/           # shared write logic (used by routes AND sim)
    posts.ts  comments.ts  likes.ts  follows.ts  notifications.ts
  sim/
    firehose.ts       # the timer engine + weighted actions
drizzle.config.ts
package.json · .env.example

Key structural decision: a services/ layer holds every state mutation. Routes are thin (validate → call service → respond), and the simulator calls the exact same services. One code path, so simulated and real data can never diverge.

Local dev with Portless

Portless serves HTTPS by default (local CA, HTTP/2), which is exactly why Secure cookies and a fixed CORS origin work locally. Use .localhost (the default) — not .local, which collides with mDNS/Bonjour.

# one-time
bun install
bun run db:migrate && bun run db:seed
portless trust                 # trust local CA (first run)

# run API at https://api.social.localhost
portless api.social bun run dev

# your frontend, separately, e.g.:
portless app.social bun run dev    # https://app.social.localhost

From the frontend, every call is fetch(url, { credentials: "include" }) and the browser handles cookies automatically — no Authorization header juggling. Swagger lives at https://api.social.localhost/docs.

Build order

Suggested milestones for implementation — each one independently runnable.

  1. Skeleton. Bun + Elysia app, config, CORS, /health, Swagger, bun:sqlite + Drizzle wired, migrations run on boot.
  2. Schema + seed. All tables & relations; faker seed incl. bot users and a starter feed.
  3. Auth. register/login/refresh/logout/me, cookie helpers, JWT guard, CSRF toggle, ownership macro.
  4. Core reads. Posts list (cursor/sort/filter), post detail (+ view increment), comments, profiles, tags — all public.
  5. Core writes. Create/edit/delete posts & comments, likes, follows, bookmarks, notifications — through the services/ layer.
  6. Realtime. WS hub + topics; wire every service mutation to broadcast post-commit.
  7. Firehose. Bot engine calling the same services; /sim/* controls.
  8. Media. Upload endpoint + static serving; bot posts get placeholder images.
Acceptance check Open /docs, subscribe a raw WS client to feed:global, start the simulator, and watch post.created events stream while new rows land in app.db — with zero authenticated clients connected. That's the whole thesis working end to end.
