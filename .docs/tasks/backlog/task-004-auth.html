Auth & sessions

Cookie-based JWT auth: register, login, refresh (with rotation), logout, me. Cookie helpers, the JWT guard, the CSRF toggle, and the ownership macro that every write/owner route will lean on.

①

Scope

  • src/lib/auth.ts — JWT sign/verify (@elysiajs/jwt), Bun.password hash/verify, opaque refresh-token generation + hashing.
  • src/lib/cookies.ts — set/clear helpers with correct flags per the design (access_token httpOnly·Secure·Lax·Path=/; refresh_token httpOnly·Secure·Lax·Path=/auth/refresh; csrf_token JS-readable).
  • src/lib/csrf.ts — double-submit check, gated by CSRF_ENABLED.
  • src/plugins/auth-guard.ts — derive user from the access cookie; .auth() macro (401 if missing/expired) and .owner() helper for ownership checks.
  • src/routes/auth.ts — POST /api/auth/register, /login, /refresh (validate vs stored hash, rotate: issue new refresh + revoke old, re-issue access), /logout (revoke + clear), GET /api/auth/me.
②

Acceptance

  • Register → 3 cookies set, user object returned.
  • Login with the seeded test user → cookies set, 200.
  • GET /me with the session → current user; without it → 401.
  • Refresh rotates the token (old refresh no longer valid) and re-issues access.
  • Logout revokes the refresh row and clears cookies.
  • With CSRF_ENABLED=true, a mutating request missing X-CSRF-Token is rejected; with the header it passes.
③

Bruno tests — 02-auth/

This folder is the keystone of the whole suite (see bruno-testing-plan.html). Requests in seq order:

  • register (seq 10) — create or 409 if exists; tolerate either since the seed already made the test user.
  • login (seq 20) — parse Set-Cookie, stash accessToken/refreshToken/csrfToken as env vars. This is what authenticates every later folder.
  • me (seq 30) — 200, returns the test user.
  • me-unauthed (seq 35) — clear cookie header, assert 401 + error.code = UNAUTHENTICATED.
  • refresh (seq 40) — 200, new cookies; re-stash.
  • logout (seq 90) — 200, then clears the env vars in post-response. Keep this last so other folders still have a session during a full run — i.e. logout lives here but the suite's write folders run before it only if ordered so; simplest is to re-login at the top of any folder that needs it, or keep logout as its own final folder. Document the choice in the collection.
Ordering note Because folders run alphabetically, 02-auth runs before write folders — good. But if logout sits in 02-auth, it would wipe auth before writes run. Resolution: put logout in its own trailing folder (e.g. 99-teardown/), or re-login via the collection pre-request if accessToken is unset. Pick one and note it in collection.bru.
