Realtime / WebSocket

One /ws endpoint, a topic hub, and every service mutation wired to broadcast after its transaction commits — so the wire is always consistent with the DB.

①

Scope

  • src/lib/realtime.ts — in-memory hub: topic → Set<socket>, with subscribe/unsubscribe/broadcast(topic, event, data).
  • Elysia .ws('/ws') — handle {action:"subscribe"|"unsubscribe", topic}; public topics open, user:{id} requires a valid access cookie at upgrade.
  • Topics + events per the design: feed:global & tag:{slug} (post.created); post:{id} (post.liked/unliked/viewed, comment.created/liked); user:{id} (notification.created).
  • Wire the seams left in task-008/009/010: each service calls hub.broadcast(...) post-commit. Server→client envelope { topic, event, data, ts }.
②

Acceptance

  • A WS client can subscribe to feed:global and receives post.created when a post is created (via REST or sim).
  • post:{id} receives like/comment/view events with fresh counts.
  • user:{id} requires auth at upgrade; rejects without a session.
  • Broadcasts fire only after the DB commit (no event for a rolled-back write).
  • Unsubscribe + disconnect clean up the hub's sets (no leaks).
③

Tests — WS script (Bruno can't do sockets)

Bruno doesn't cover WebSockets, so ship a tiny scripts/ws-smoke.ts run with bun: open a socket, subscribe to feed:global, create a post via fetch (with the auth cookie), and assert a post.created event arrives within a timeout (exit non-zero on failure). Document bun run scripts/ws-smoke.ts in the README. REST regressions still go through the existing Bruno folders.
