Firehose simulator

The payoff: bot users that post/comment/like/follow on a timer through the same services real requests use — so the feed visibly streams with zero authenticated clients. Plus the /sim/* control surface.

①

Scope

  • src/sim/firehose.ts — a timer firing every SIM_INTERVAL_MS; each tick runs SIM_ACTIONS_PER_TICK weighted random actions as is_bot users via the services layer (create post 20% · comment 35% · like post 30% · like-comment/follow 15%). Bot posts use placeholder image URLs. Auto-starts when SIM_ENABLED=true.
  • src/routes/sim.ts (gated by SIM_CONTROL_ENABLED):
    • GET /api/sim/status — running state + config.
    • POST /api/sim/start · POST /api/sim/stop.
    • PATCH /api/sim/config — interval / actions-per-tick / weights.
  • Because bots go through the services, every action broadcasts the same WS events and lands in SQLite identically to real writes.
②

Acceptance — the whole thesis

  • Start the sim → new posts/comments/likes appear in app.db over time.
  • A raw WS client on feed:global sees post.created stream with zero authenticated clients connected.
  • /sim/status reflects running state; start/stop toggle it; config patch changes cadence/weights live.
  • Counts stay consistent (bots use the same transactional services).
  • /sim/* is hidden when SIM_CONTROL_ENABLED=false.
③

Bruno tests — 10-sim/

Public control endpoints. status → assert shape; start → status shows running; config patch → reflected in status; stop → not running. For the streaming proof, reuse the scripts/ws-smoke.ts from task-011 but driven by the sim instead of a manual POST: start sim, assert a post.created arrives, stop sim. Document the full acceptance check (open /docs, subscribe WS, start sim, watch the stream) in the README.
