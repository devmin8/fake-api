Backlog ticket · task-005

Posts: reads & pagination

The cursor engine plus the public post reads: list (sort/filter/ directional cursor), detail (with view increment), and the cheap "how many newer" head-check.

①

Scope

  • src/lib/cursor.ts — encode/decode the opaque base64 created_at|id cursor; the directional older|newer contract from the design's Conventions (newer = ascending-from-cursor, adjacent-to-head first).
  • Collection envelope helper: { data, newestCursor, oldestCursor, hasMore }.
  • src/routes/posts.ts (reads only this task):
    • GET /api/posts — ?sort=new|top|trending, ?tag=, ?author=, ?cursor=, ?dir=older|newer, ?limit= (default 20, max 100). top/trending paginate older-only within a frozen order.
    • GET /api/posts/:id — detail; increments view_count (broadcast wired later in task-011).
    • GET /api/posts/updates?since=<cursor> → { count, newestCursor, previews? }.
②

Acceptance

  • List returns the envelope with both boundary cursors and correct hasMore.
  • dir=older backfills strictly older; dir=newer returns strictly newer, adjacent-to-head first. No gaps/dupes across two stitched pages.
  • sort=top orders by like_count; trending blends likes+comments decayed by age.
  • Filters ?tag= and ?author= narrow correctly.
  • Detail increments view_count by 1 per call.
  • /updates returns an accurate count of newer items.
③

Bruno tests — 03-posts-read/

All public (no auth needed). Cover: default list shape; limit respected; a two-page older backfill using the returned oldestCursor; a newer catch-up; sort=top; tag + author filters; detail 200 + capture postId into an env var for reuse; detail 404 on a bad id; /updates count. Capture a real id from the list response rather than hardcoding.
