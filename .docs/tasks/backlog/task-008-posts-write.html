Posts & comments: writes

The services/ layer begins here. Create/edit/delete posts and create/delete comments — through shared write services so the firehose can reuse the exact same code path later.

①

Scope

  • src/services/posts.ts — createPost, updatePost, deletePost (cascades). Maintains tags via post_tags.
  • src/services/comments.ts — createComment (bumps the post's comment_count in the same transaction), deleteComment.
  • Routes (writes added to existing files):
    • POST /api/posts AUTH — title, body, optional image, tags.
    • PATCH /api/posts/:id OWNER · DELETE /api/posts/:id OWNER.
    • POST /api/posts/:id/comments AUTH.
    • DELETE /api/comments/:id OWNER.
  • Validation via Elysia t; errors through the standard envelope. Broadcasts are stubbed/no-op until task-011 (leave a clean seam — services return enough for a broadcaster to fire later).
②

Acceptance

  • Create post returns the row; it appears in GET /posts.
  • Edit/delete enforce ownership (403 for a non-owner; 401 unauthenticated).
  • Delete cascades (comments/likes/post_tags gone).
  • Comment create bumps comment_count; delete restores it.
  • Invalid body → 422 + VALIDATION_ERROR.
  • Counts stay consistent under create→delete.
③

Bruno tests — 06-posts-write/

Auth required — relies on 02-auth/login having set the auth env vars. Chain (seq): create post → capture postId → it shows in list → patch it → comment on it (capture commentId) → delete the comment → delete the post. Boundary cases: create unauthenticated → 401; patch/delete as a second user → 403 (register a throwaway user for this, or assert with no session); bad body → 422.

Second-user ownership test To test 403 cleanly, add a register for a second user in this folder, swap the auth env vars to that user for the owner-violation request, then restore the primary user's vars. Document the swap in the request's pre/post scripts.
