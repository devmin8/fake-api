Follows, feed & notifications

The social graph closes: follow/unfollow, the personalized /feed, profile self-edit, and the notifications that get written whenever someone likes/comments/follows you.

①

Scope

  • src/services/follows.ts — follow/unfollow; writes a notification.
  • src/services/notifications.ts — createNotification used by likes/comments/follows services (retro-wire task-008/009 calls to emit notifications).
  • Routes:
    • POST/DELETE /api/users/:username/follow AUTH.
    • GET /api/feed AUTH — posts from people you follow, cursor paginated.
    • PATCH /api/users/me AUTH — display name, bio, avatar.
    • GET /api/notifications (?unread=true) AUTH.
    • POST /api/notifications/:id/read and /read-all AUTH.
②

Acceptance

  • Follow → followee gains a follower; a notification row is written.
  • /feed returns only posts from followed users, paginated; empty when you follow nobody.
  • Liking/commenting (from task-008/009) now writes a notification to the recipient.
  • PATCH /users/me updates only allowed fields.
  • ?unread=true filters; mark-read and read-all flip is_read.
  • All endpoints 401 unauthenticated; can't follow yourself (guard).
③

Bruno tests — 08-follows-feed-notifications/

Auth required. Follow a seeded user → assert their follower count rose and a notification exists for them (or assert via the actor side). Fetch /feed; after following someone with posts, the feed is non-empty. Patch /users/me and read it back. List notifications, mark one read, then read-all → all is_read=true. 401 + self-follow 409/422 cases.
