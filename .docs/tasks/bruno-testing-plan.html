Testing strategy · read before writing any Bruno tests

Bruno Testing Plan

How we test every API as we build it. The goal: tests that run in a deterministic order, log in once, and stay green in both the Bruno app and the headless bru CLI.

01

Why this plan exists

Cookie + CSRF auth is the thing that bites you when every request needs a session. Two facts drive the whole design:

  1. The Bruno desktop app has an automatic cookie jar — Set-Cookie is captured and replayed on later requests. Interactively, auth "just works" after login.
  2. The Bruno CLI does not auto-save Set-Cookie from 2xx responses (only from 3xx redirects — Bruno issues #8090/#6000). In CI the jar is empty and nothing is authenticated unless we capture the cookie ourselves.

So we capture auth explicitly on login and re-inject it via a collection-level pre-request script. That one pattern makes the suite behave identically in the app and the CLI.

02

Collection layout

The Bruno collection lives at the repo root in bruno/. One numerically-prefixed folder per resource so run order is deterministic (the runner orders folders alphabetically, and requests within a folder by their seq).

bruno/
  bruno.json                 # collection config (scripts.flow = sandwich)
  collection.bru             # collection vars + pre-request: inject auth + CSRF
  environments/
    local.bru                # baseUrl + test credentials
  01-health/
  02-auth/                   # register/login/me/refresh/logout — runs FIRST
  03-posts-read/
  04-comments-read/
  05-users-tags-read/
  06-posts-write/
  07-likes-bookmarks/
  08-follows-feed-notifications/
  09-media/
  10-sim/
  • Folders: numeric prefix (01-, 02-, …) = alphabetical = intended order.
  • Requests: explicit seq in meta, leaving gaps (10, 20, 30) so you can insert later without renumbering.
  • 02-auth/login must run before any folder that needs a session.
03

Auth strategy (the important part)

1 · Login captures the cookies into environment vars

Login (and register) parse Set-Cookie off the response headers and store the raw values so we can replay them. We use env vars (persist for the whole run), not runtime vars.

# bruno/02-auth/login.bru
meta {
  name: Login
  type: http
  seq: 20
}
post {
  url: {{baseUrl}}/api/auth/login
  body: json
  auth: none
}
body:json {
  { "email": "{{testEmail}}", "password": "{{testPassword}}" }
}
script:post-response {
  // Works in BOTH app and CLI: read Set-Cookie straight off the
  // response, don't depend on the jar being populated.
  const raw = res.headers['set-cookie'] || [];
  const cookies = Array.isArray(raw) ? raw : [raw];
  const pick = (name) => {
    const hit = cookies.find(c => c.startsWith(name + '='));
    return hit ? hit.split(';')[0].split('=').slice(1).join('=') : null;
  };
  const access  = pick('access_token');
  const refresh = pick('refresh_token');
  const csrf    = pick('csrf_token');
  if (access)  bru.setEnvVar('accessToken', access);
  if (refresh) bru.setEnvVar('refreshToken', refresh);
  if (csrf)    bru.setEnvVar('csrfToken', csrf);
}
assert {
  res.status: eq 200
}

2 · Collection pre-request replays cookie + CSRF on every request

# bruno/collection.bru
script:pre-request {
  const access  = bru.getEnvVar('accessToken');
  const refresh = bru.getEnvVar('refreshToken');
  const csrf    = bru.getEnvVar('csrfToken');
  // Re-send the auth cookie ourselves (CLI jar won't).
  const parts = [];
  if (access)  parts.push(`access_token=${access}`);
  if (refresh) parts.push(`refresh_token=${refresh}`);
  if (csrf)    parts.push(`csrf_token=${csrf}`);
  if (parts.length) req.setHeader('Cookie', parts.join('; '));
  // Double-submit CSRF header for mutating requests.
  if (csrf) req.setHeader('X-CSRF-Token', csrf);
}

Every request inherits this, so individual requests don't repeat auth wiring — they just assume they're logged in. Public requests are unaffected: an extra cookie/header is harmless.

Tip while learning a frontend The API gates CSRF behind CSRF_ENABLED. Run the test API with CSRF_ENABLED=false to drop the header dance entirely. Keep the capture code anyway, so turning CSRF back on needs no test changes.

3 · Logout / refresh

  • refresh is the only request needing the refresh cookie — the collection pre-request already injects it.
  • logout clears the env vars in its post-response so later runs start clean.
04

Capturing ids for chained requests

Requests that create rows hand ids to later requests. Capture into an env var in the post-response and reference with {{…}}:

script:post-response {
  bru.setEnvVar('postId', res.body.id);
}

A later request then uses {{baseUrl}}/api/posts/{{postId}}. This is how "create post" → "like post" → "comment" → "delete post" chains inside a write folder without hardcoding ids.

05

Assertions

  • Use the declarative assert block for cheap status/shape checks.
  • Use script:tests (Chai expect) only when you need logic (loops, computed values, conditionals).
  • Every request gets at least a status assertion. Error cases get their own request asserting the envelope res.body.error.code.
assert {
  res.status: eq 200
  res.body.id: isDefined
  res.body.data: isArray
}
06

Per-resource coverage checklist

Each build task adds its folder and covers, at minimum:

  • Happy path for every endpoint it introduces.
  • Auth boundary: one unauthenticated call to an AUTH endpoint asserting 401.
  • Ownership boundary (where relevant): a second user hitting an OWNER endpoint asserting 403.
  • Not found: a bad id asserting 404.
  • Validation: a bad body asserting 422 + error.code = VALIDATION_ERROR.
07

Running headless (CI / quick check)

npm install -g @usebruno/cli      # or: npx @usebruno/cli
cd bruno
bru run --env local                       # whole suite, in order
bru run 06-posts-write --env local        # one folder
bru run --env local --reporter-junit out/results.xml

Non-zero exit on any failed assertion, so it gates a pipeline directly. Folders run alphabetically and requests by seq, so 02-auth logs in before the write folders.

08

Gotchas (don't relearn these)

  1. CLI won't save 2xx Set-Cookie — we capture + replay manually. Never assume the jar.
  2. jar.getCookie(…) is async — but we avoid the jar entirely and read res.headers['set-cookie'], which works everywhere.
  3. Folders order alphabetically — keep the numeric prefixes.
  4. Leave seq gaps within a folder.
  5. WebSocket topics aren't covered by Bruno — the realtime task ships a tiny bun script that subscribes and asserts an event instead.
